4-5 Obtaining new instances of prototype beans inside singleton beans

4-5 Obtaining new instances of prototype beans inside singleton beansMotivationApplicationContextAware InterfaceFunctionLifecycle IntefaceHow to use<lookup-method> elementFunctionMethod InjectionHow to useExampleRestrictions<replaced-method> elementFunctionExampleUniquely identifying the bean method

Motivation

  • The prototype-scoped dependency of a singleton bean is injected at the time of creation of the singleton bean.

  • The Spring container creates instance of a singleton bean only once, which means the singleton bean holds reference to the same prototype bean instance throughout its lifetime.

  • In this section, we provide three methods for a singleton bean to retrieve a new instance of their prototype-scoped dependency from the Spring container:

    • Make the singleton bean's class implement Spring's ApplicationContextAware interface.
    • Use the <lookup-method> element of Spring's beans schema.
    • Use the <replace-method> element of Spring's beans schema.

 

ApplicationContextAware Interface

Function
  • Spring's ApplicationContextAware interface is implemented by beans that require access to the ApplicaationContext instance in which they are running.
  • The ApplicationContextAware interface defines a single method, setApplicationContext, which provides the implementing beans with an instance of the ApplicationContext object. This method is called by the Spring container at the time of bean creation.
Lifecycle Inteface

  • ApplicationContextAware interface is a lifecycle interface.

  • A lifecycle interface defines one or more callback methods that are invoked by the Spring container at the appropriate times during the bean's lifetime.

    • For example, the setApplicationContext method is called by the Spring container after the bean instance is created but before the bean instance is completely initialized.
How to use
  • A bean that implements the ApplicationContextAware interface can access other bean registered with the ApplicationContext instsance by calling ApplicationContext's getBean() method.
  • This means that if a singleton bean implements the ApplicationContextAware interface, it can explicitly obtain its prototype-scoped dependency by calling ApplicationContext's getBean method.

 

<lookup-method> element

Function
  • If a bean class defines a bean lookup method whose return type represents a bean, the <lookup-method> element instructs Spring to provide implementation for this method.
  • The method implementation provided by Spring is responsible for retrieving the bean instance from the Spring container and returning it.
Method Injection
  • The use of <lookup-method> element to lookup beans is referred to as a method injection technique, because the <lookup-method> element injects a bean lookup method implementation into the bean class.
How to use

  • The <lookup-method> element's bean attribute specifies the name of the bean to be looked-up, and the name attribute specifies the name of the method whose implementation is to be provided by Spring.

    • In short, <lookup-method> instructs Spring to implement the <name> method and returns a <bean> bean.
    • Note that, the bean lookup method defined by the bean class can be abstract or a concrete method. In the example below, we could have defined the getCustomerRequestDetails method as a concrete method. As the getCustomerRequestDetails method is overridden by Spring, it doesn't matter if we perform any action or leave it empty.

  • The bean definition for CustomerRequestServiceImpl class contains a <lookup-method> element, which implements getCustomerRequestDetails and retrieve a customerRequestDetails bean.

    • name: instructs Spring to provide implementation for the getCustomerRequestDetails lookup method
    • bean: the implementation of getCustomerRequestDetails retrieves a bean with id as customerRequestDetails from the container and returns it.

  • In the Java code, the submitRequest method calls the getCustomerRequestDetails bean lookup method to obtain a CustomerRequestDetails instance. The scope="prototype" instructs Spring to retrieve a new instance of CustomerRequestDetails every time the method is invoked.

Restrictions

  • The following restrictions apply to the signature of the bean lookup method:

    1. The method must be defined as public or protected.
    2. The method must not accept any argument.
    3. The bean class and bean lookup method must not be defined as final (otherwise Spring won't be able to implement it at runtime).

 

<replaced-method> element

Function
  • The <replaced-method> element allows you to replace any arbitrary method (abstract or concrete) in a bean class with a different implementation.
  • We need to provide information about the overridden method and the overriding method.

  • Spring's MethodReplacer interface defines a reimplement method whose implementation is provided by the MyMethodReplacer class. The reimplement method represents the overriding method.

  • We let MyMethodReplacer class to implement Spring's ApplicationContextAware interface so that the reimplement method can use the ApplicationContext instance to obtain bean instances from the Spring container.

  • The reimplement method accepts the following arguments:

    • Object obj - the object (class) whose method we are overriding. In our case, it is the CustomerRequestServiceImpl object.
    • Method method - the bean method that is overridden by the reimplement method. In our case, it is the getMyBean method.
    • Object[] args - arguments passed to the bean method being overridden. In our case, it is the String beanName.

  • The reimplement method of class MyMethodReplacer uses the args argument to obtain the bean name passed to the CustomerRequestServiceImpl's getMyBean method, then calls ApplicationContext's getBean method to obtain the corresponding bean instance. This allows the runtime call to getMyBean return the correct bean instance we were looking for.
